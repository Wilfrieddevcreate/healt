# Deploying FitHorizon to Render

This project ships with a `render.yaml` Blueprint so you can deploy the full stack (web service + persistent disk for the SQLite database) with a single click.

## Prerequisites

- A [Render account](https://dashboard.render.com/)
- This repository pushed to GitHub (already done)
- A [Groq API key](https://console.groq.com) for AI article generation

## One-click deploy (Blueprint)

1. Go to https://dashboard.render.com/blueprints
2. Click **New Blueprint Instance**
3. Connect your GitHub account and select `Wilfrieddevcreate/healt`
4. Render will detect `render.yaml` and show a preview of the service
5. Click **Apply**
6. Set the environment variables that are marked `sync: false`:
   - `GROQ_API_KEY` — your Groq API key (required for AI features)
   - `NEXT_PUBLIC_GOOGLE_VERIFICATION` — Google Search Console code (optional)
   - `NEXT_PUBLIC_BING_VERIFICATION` — Bing Webmaster code (optional)
   - `NEXT_PUBLIC_ADSENSE_ID` — your AdSense client ID (optional)
7. Click **Create Service**

Render will build the Docker image, attach a 1 GB persistent disk at `/app/data`, run migrations automatically on startup, and deploy the service.

## First-time setup (after the service is live)

Once the service is up, seed the database with the initial admin user + categories + sample posts.

First, grab the `SEED_TOKEN` value from the Render dashboard (Environment tab), then:

```bash
curl https://fithorizon.onrender.com/api/seed?token=YOUR_SEED_TOKEN
```

The seed endpoint is idempotent — it skips if posts already exist, so it's safe to hit multiple times.

Then log in to the admin at `/admin/login`:

- Email: `admin@fithorizon.com`
- Password: `admin123`

**Change the admin password immediately** by updating the seed or directly in the database.

## Custom domain

1. In the Render dashboard, open your service → **Settings** → **Custom Domains**
2. Add your domain (e.g. `fithorizon.com`) and follow the DNS instructions
3. Update the `NEXT_PUBLIC_SITE_URL` env var to your custom domain
4. Redeploy (Render does this automatically when env vars change)

## Environment variables reference

| Variable | Required | Description |
|----------|----------|-------------|
| `DATABASE_URL` | Yes | Set automatically to `file:/app/data/prod.db` by the Blueprint |
| `JWT_SECRET` | Yes | Auto-generated by Render on first deploy |
| `NEXT_PUBLIC_SITE_URL` | Yes | The canonical URL of your site (used for SEO + OG tags) |
| `GROQ_API_KEY` | Optional | Required for AI article generation in admin |
| `NEXT_PUBLIC_GOOGLE_VERIFICATION` | Optional | Google Search Console verification code |
| `NEXT_PUBLIC_BING_VERIFICATION` | Optional | Bing Webmaster Tools verification code |
| `NEXT_PUBLIC_ADSENSE_ID` | Optional | Google AdSense client ID (e.g. `ca-pub-XXXXX`) |

## Plan & pricing

The Blueprint uses the `starter` plan ($7/month) because:

- The **free** plan spins down after 15 min of inactivity. Cold starts take 30–60s which is fatal for SEO (Google will flag you as slow).
- The starter plan is always-on.
- The 1 GB persistent disk costs $0.25/month extra.

**Total: ~$7.25/month** for a production-ready, SEO-optimized blog.

If you want to save money, you can switch to the free plan in `render.yaml` (`plan: free`) but expect cold starts.

## Scaling beyond SQLite

SQLite on a single instance works well for tens of thousands of articles and millions of monthly visitors. If you need horizontal scaling:

1. Create a Render Postgres instance
2. Update `prisma/schema.prisma` datasource to `provider = "postgresql"`
3. Remove the `@prisma/adapter-libsql` code in `lib/prisma.ts` and use the default Postgres connection
4. Update `DATABASE_URL` env var to the Postgres connection string
5. Run `npx prisma migrate dev` locally to regenerate migrations
6. Redeploy

## Monitoring

- **Logs**: Render dashboard → your service → **Logs**
- **Metrics**: Render dashboard → your service → **Metrics**
- **Database**: SSH into the container via Render Shell and inspect `/app/data/prod.db` with `sqlite3`

## Troubleshooting

**Build fails with "no such table":** You're probably building without the migration step. The `Dockerfile` runs `prisma migrate deploy` at both build time (to satisfy SSG) and runtime (to sync the persistent disk). Make sure your `render.yaml` points to the `Dockerfile` at the repo root.

**Admin login doesn't work after deploy:** The database is empty. Run `curl https://<your-domain>/api/seed` once.

**"Service Unavailable" on first request after 15 min:** You're on the free plan. Upgrade to starter in `render.yaml`.
